# Resume Builder V2.3 Data Recovery Center Design

Date: 2026-07-11

## Goal

V2.3 在现有本地编辑器中增加“恢复历史数据”能力，让用户可以从导入前和恢复前的整套数据快照中选择一个时间点恢复。恢复必须保留当前数据、保留被选中的历史快照，并在失败时自动回滚，不改变一页 A4、实时草稿、显式保存、正式生成和外部私有数据目录等既有行为。

## Product Decisions

- 入口位于左侧预览顶部的 `数据管理` 菜单，新增 `恢复历史数据`。
- 恢复中心同时展示 `.pre-import-*` 与 `.pre-restore-*` 快照，按时间从新到旧排列。
- 被恢复的历史快照保持不变，可以重复恢复。
- 每次恢复前，当前完整数据自动保存为新的 `.pre-restore-*` 快照，因此恢复操作本身可撤销。
- 当前存在未保存修改时禁止恢复，用户必须先保存或放弃修改。
- V2.3 只做整套数据恢复，不做单份简历恢复、内容合并、快照删除或重命名。

## User Experience

点击 `数据管理 -> 恢复历史数据` 后，编辑器请求快照列表并打开现有数据管理弹窗。每条记录显示快照类型、创建时间、有效状态、简历数量、活动简历和简历名称。有效记录可以选择；无效记录保留在列表中但禁用恢复，并显示不包含本机绝对路径的简短原因。没有快照时显示空状态，不显示可执行的恢复按钮。

选择有效记录后进入二次确认状态，明确说明恢复会整套替换当前已保存的简历、照片、备份和配置，当前数据会先自动保存。确认后按钮进入不可重复提交的加载状态。成功时关闭弹窗、应用新的注册表、重新读取活动简历并显示草稿预览；正式 PDF 状态回到待生成。失败时弹窗保持打开，当前正式数据保持原样，并显示可操作错误。

## Snapshot Discovery And Security

恢复模块只扫描 `dataRoot` 的父目录，并只接受当前数据目录 basename 对应的精确名称：

```text
<dataRoot basename>.pre-import-YYYYMMDD-HHMMSS[-N]
<dataRoot basename>.pre-restore-YYYYMMDD-HHMMSS[-N]
```

任意其他同级目录都被忽略。根目录或目录树中出现符号链接、特殊文件或不安全结构时，该候选快照标记为无效。有效候选必须通过现有 `validateDataRoot()`，包括注册表、YAML、照片引用和路径边界检查。

前端永远不提交文件路径。服务端根据候选目录 basename 计算稳定的 SHA-256 快照 ID；列表接口只返回 ID 和摘要。恢复请求到达后服务端重新扫描并通过 ID 查找候选，防止使用过期结果、路径穿越或伪造任意路径。接口响应和浏览器错误不暴露私有数据目录的绝对路径。

## Atomic Restore And Rollback

恢复由独立的 `scripts/data-recovery.mjs` 管理，流程如下：

1. 服务端在调用恢复管理器前获取整套数据替换门禁；已有写操作未结束时返回 HTTP 423。
2. 服务端保持门禁，并由管理器重新发现目标快照、确认其仍然有效。
3. 在门禁内将目标快照复制到同一父目录下、由当前进程拥有的临时恢复目录，并校验副本完整性。
4. 将当前 `dataRoot` 原子重命名为唯一的 `.pre-restore-<timestamp>`。
5. 将临时恢复目录原子重命名为正式 `dataRoot`。
6. 再次校验已发布目录。
7. 如果发布或最终校验失败，隔离失败目录并将 `.pre-restore-*` 恢复为正式 `dataRoot`；管理器返回后服务端才释放门禁。

原始历史快照不参与重命名，因此成功、失败和进程中断都不会消耗唯一副本。失败或未发布的 staging 不会被递归自动删除，而是保留在数据目录旁供检查，因此多次失败后可能累积。`dispose()` 只释放进程内状态，不执行 staging 清理，更不会删除正式 `dataRoot`、`.pre-import-*` 或 `.pre-restore-*`。

## Concurrency And APIs

新增本地接口：

```text
GET  /api/data/recovery/snapshots
POST /api/data/recovery/restore
```

列表接口返回快照摘要；恢复接口只接收 `{ snapshotId }`。现有导入提交和新恢复提交共用一个“整套数据替换”门禁：任一操作等待或执行时，新的保存、生成、照片上传、简历管理和另一项替换操作都会被拒绝。已有写操作仍在执行时，替换请求返回 423，用户可以稍后重试。读接口保持现有行为，但导出不会在目录替换期间运行。

V2.3 的并发边界是同一 `dataRoot` 只运行一个编辑器进程。门禁和恢复中的 `restore-locked` 只协调当前服务器进程内的请求；实现不创建跨进程锁文件，也没有过期锁回收或手动清锁步骤。启动第二个指向同一 `dataRoot` 的编辑器进程不受支持，恢复前必须先关闭它。

典型错误状态为：400 无效请求、404 快照不存在、409 快照已失效或无效、423 数据写入/替换正在进行。恢复模块将底层异常转换为稳定错误代码和不包含绝对路径的安全摘要，UI 再映射为对应的中文提示。

## Testing

- 单元测试覆盖名称匹配、时间解析与排序、两种快照类型、稳定 ID、无效数据、符号链接和任意目录忽略。
- 事务测试覆盖成功恢复、原快照保留、唯一 `.pre-restore-*`、复制失败、失败 staging 保留、发布失败、最终校验失败和自动回滚。
- HTTP 测试覆盖列表、恢复、伪造 ID、失效快照、423 门禁、管理器释放和不返回绝对路径。
- 浏览器测试覆盖入口、空状态、无效记录、详情与确认、未保存修改拦截、重复提交保护、成功后注册表/草稿刷新。
- 完整 `npm test` 必须继续通过导入导出、生成、隐私扫描及现有编辑器回归测试。

## Non-Goals

V2.3 不提供快照删除、自动过期清理、容量配额、快照重命名、单份简历恢复、差异比较、数据合并、ZIP 恢复入口、云同步、账号、多设备自动发现或正式输出恢复策略。有限排版控制继续留到 V2.3 完成之后评估。
