# Resume Builder V2.3 Data Recovery Boundaries

Date: 2026-07-11

## Goal And User Workflow

V2.3 在本地编辑器中增加整套数据恢复中心，用于把当前数据恢复到一次导入前或恢复前的本机历史状态。

1. 打开 `数据管理 -> 恢复历史数据`。
2. 查看按时间倒序排列的有效与无效快照，选择一条有效记录并检查简历数量、活动简历和名称详情。
3. 二次确认整套替换。当前存在未保存草稿时必须先保存或撤销，不能开始恢复。
4. 恢复成功后编辑器应用恢复 API 返回的注册表，再读取活动简历及其单份简历备份。读取成功后，现有预览流程异步安排草稿渲染。快照可能同时带回旧 PDF、PNG 和 HTML，但界面不会把它们视为当前输出；需要显式点击 `生成 PDF`。

## Snapshot Scope And Semantics

恢复中心只发现当前 `dataRoot` 的同级目录，且名称必须精确匹配：

```text
<dataRoot>.pre-import-YYYYMMDD-HHMMSS[-N]
<dataRoot>.pre-restore-YYYYMMDD-HHMMSS[-N]
```

- `pre-import` 是整套数据导入提交前保留的旧数据。
- `pre-restore` 是整套数据恢复提交前保留的旧数据。
- 可选的 `-N` 是同一秒发生名称冲突时使用的唯一序号；现有目录不会被覆盖。
- 其他同级目录、名称不完整、时间无效或 basename 不匹配的目录不会进入列表。

恢复按完整数据目录处理，替换注册表、全部简历 YAML、照片、简历备份和目录内其他已保存数据，不做按简历合并。快照中的 `output/<id>/` 也可能随完整目录被物理发布，但 V2.3 不保证其中的 PDF、PNG 和 HTML 与恢复后的内容匹配，也不把它们标记为当前输出。它与页面底部的 `最近备份` 不同：`最近备份` 只恢复当前单份简历的一份 YAML，恢复中心恢复整个数据集及活动简历状态。

## Discovery And Security Boundary

- 候选目录必须是精确的普通文件树；根目录或内部出现符号链接、特殊文件或不安全结构时会列为无效，且不会被遍历恢复。
- 有效快照还必须通过现有数据根校验，包括注册表、YAML、照片引用和真实路径边界规则。
- 无效快照继续显示，但不可选择；界面只显示固定、可操作且不含路径的原因。
- 浏览器只接收并提交不透明快照 ID，不提交文件路径。服务端恢复前重新扫描并通过 ID 解析候选，拒绝伪造、消失或已经失效的记录。
- API、浏览器提示和成功消息不得暴露外部数据目录的绝对路径；成功消息最多显示快照或恢复前备份的 basename。

## Restore Transaction

服务端在调用恢复管理器前先取得同一进程内的整套数据替换门禁，并在以下全部步骤完成或失败返回后才释放：

1. 服务端重新发现并验证选中的源快照。
2. 把源快照复制到同级、由当前恢复进程拥有的 staging 目录，核对精确文件指纹并再次验证。
3. 把当前 `dataRoot` 原子重命名为唯一的 `pre-restore` 快照。
4. 把 staging 原子发布为新的 `dataRoot`，然后执行最终验证。
5. staging 发布重命名失败时原地保留未发布 staging 并回滚旧数据；已经发布的数据在身份、内容指纹或最终校验中失败时才隔离该目录并回滚旧数据。无法自动隔离或回滚时，保留所有仍存在的数据根并返回明确的人工恢复错误，不删除幸存副本。

被选中的源快照只用于复制，不参与重命名或删除。成功恢复、失败恢复和重复恢复都保留源快照，因此同一快照可以再次使用。

失败或未发布的 staging 不做递归自动删除，而是保留在数据目录旁供检查，因此重复失败后可能累积。系统没有 staging 自动清理错误，也不会在服务器退出时尝试删除这些目录。

## Concurrency And Drafts

- 同一个 `dataRoot` 只支持一个编辑器进程。V2.3 不创建跨进程锁文件，不提供过期锁回收或手动清锁；恢复前必须关闭仍指向该目录的其他编辑器进程。
- 同一服务器内，导入提交和恢复提交共用整套数据替换门禁；服务端在调用恢复管理器之前取得门禁，并在包含 staging 复制、验证、发布和回滚的整个恢复调用结束后才释放。保存、生成、照片上传、简历管理、导出或另一项替换不能与其交叠。
- 已有写入未结束时恢复返回可重试的忙碌状态，不抢占或绕过正在执行的写入。
- 未保存草稿会在打开恢复中心和最终提交前再次拦截，避免内存中的修改被整套替换丢弃。

## Privacy Boundary

活动 `dataRoot` 必须位于代码仓库之外的私有外部位置；同级的 `pre-import` 和 `pre-restore` 快照也必须保持在 Git 之外。快照是未加密的个人数据副本，可能包含姓名、联系方式、简历正文、照片和历史备份，应只保存在可信设备和目录中。

快照位于活动 `dataRoot` 之外，不会包含在跨电脑导出的 ZIP 中，也不会自动同步到另一台电脑。ZIP 迁移只传输活动数据包；目标电脑只能发现其本机数据根旁已经存在的历史快照。

## Error And Recovery Behavior

- 列表读取失败时可重试读取，不会修改数据。
- 选中快照消失或变为无效属于终态不可用：界面重新读取列表、清除旧选择，并要求重新选择。
- 同服务器内恢复或写入门禁繁忙属于暂时错误：确认详情和选择保持可用，用户可以稍后重试恢复。
- 恢复提交前失败时当前正式数据保持不变；发布后校验失败时执行隔离和回滚。
- 恢复已经提交后，界面先应用恢复 API 返回的注册表；只有随后读取活动简历或该简历备份失败时，恢复对话框才明确提示“数据已恢复”并提供 `重试刷新`。
- `重试刷新` 只重复读取活动简历及其单份简历备份，不会请求 `/api/resumes`，也不会再次提交恢复。
- 上述读取成功后恢复对话框关闭，现有预览流程再异步安排草稿渲染。草稿渲染失败使用已有的 `草稿预览失败` 状态，不属于恢复刷新失败，也不会保持或重新打开恢复对话框。
- 恢复完成后正式输出状态为待生成。即使快照物理带回了 PDF、PNG 或 HTML，它们也不代表当前恢复内容，必须显式重新生成。

## Explicit Non-Goals

V2.3 不提供：

- 快照删除、自动清理、过期策略、容量配额或重命名。
- 单份简历恢复、差异比较、内容合并或冲突解决。
- 云同步、账号、多设备自动发现或远程恢复。
- 通过恢复中心导入 ZIP，或把本机历史快照加入 ZIP 导出。
- 对快照中物理恢复的 `output/` 提供正式输出有效性保证或恢复策略。
- 绕过未保存草稿保护，或在个人运行数据上人工制造事务失败。
